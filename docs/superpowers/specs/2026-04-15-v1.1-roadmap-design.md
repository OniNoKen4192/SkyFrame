# SkyFrame v1.1 Roadmap

**Date:** 2026-04-15
**Status:** Draft — pending user review
**Source ideas:** [docs/userInput/v1.1 ideas.txt](../../userInput/v1.1%20ideas.txt)

## Goals

Ship six incremental v1.1 enhancements in an order that minimizes refactoring per step. Each step is small enough to ship as its own PR (or grouped naturally), and no step requires undoing work from an earlier step.

## Hard constraints (preserved from v1.0)

These remain non-negotiable from [CLAUDE.md](../../../CLAUDE.md):

- No ads, analytics, telemetry, or third-party trackers.
- No API keys, no account-gated providers. NWS only.
- No transmitted data beyond what is needed to fetch the forecast.
- Minimize dependencies.

## Spec landmark

CLAUDE.md's "Locked technical decisions" section lists the hardcoded location ({CITY}, ZIP {ZIP}, lat/lon, MKX grid, KMKE station) under product-requirement rules. Step 6 below intentionally relaxes that lock now that v1.0 has shipped. CLAUDE.md will be updated as part of Step 6 to reflect the new model (location becomes user-configurable).

## Sequence overview

| # | Feature | PR grouping | Estimated effort |
|---|---|---|---|
| 1 | Offline indicator | Standalone PR | ~1-2 hours |
| 2 | Tabbed views + switcher buttons | Standalone PR (ideas #4 + #5 grouped — useless apart) | ~2-3 hours |
| 3 | Alerts fetch + severe alert banner | Standalone PR | ~half day |
| 4 | Alert-driven UI color override | Standalone PR (includes scoped CSS-var refactor) | ~half day |
| 5 | Settings gear (°F/°C + curated color picker) | Standalone PR, may split into two if it grows | ~full day |
| 6 | Location change | Deferred to its own brainstorm; in v1.1 scope | TBD |

Each step builds on the previous without rework. Steps 1–2 are pure UI with zero shared infrastructure. Step 3 introduces alert plumbing that Step 4 reuses. Step 4 introduces CSS variables that Step 5 reuses.

## Step 1 — Offline indicator (idea #1)

**Goal:** Make it visually obvious when the client cannot reach the server.

**Behavior:**
- When a fetch to `/api/weather` fails, the Footer's `dot` element pulses and turns red.
- The station name in the Footer is replaced with the literal text `OFFLINE`.
- On the next successful fetch, both revert to the normal display.

**Implementation surface:**
- [client/src/components/Footer.tsx](../../../client/src/components/Footer.tsx): conditionally render `OFFLINE` text and pass an offline class to the dot.
- [client/src/styles.css](../../../client/src/styles.css) (or wherever the dot styles live): add `dot-offline` class with red color and pulse keyframes.

**Open question for implementation:** trigger on the *first* failure or only after N consecutive failures? First-failure is simpler but may flicker on transient network blips. Default: trigger on first failure since the client retries every 30s on error — flicker would be brief and informative.

**Out of scope:** any retry behavior change. The existing 30s-on-error retry stays as-is.

## Step 2 — Tabbed views + switcher buttons (ideas #4 + #5)

**Goal:** Replace the current always-stacked three-panel layout with a tabbed view: only one panel visible at a time, switched via buttons in the TopBar.

**Behavior:**
- Default view on load: Current.
- TopBar shows three buttons: **Current** | **Hourly** | **Outlook**.
- Clicking a button shows that panel and hides the other two.
- Active tab is visually distinct.
- View selection is *not* persisted across reloads (stays default = Current). If the user later wants persistence, that's a small addition during Step 5.

**Implementation surface:**
- [client/src/App.tsx](../../../client/src/App.tsx): add `activeView` state (`'current' | 'hourly' | 'outlook'`), conditionally render one panel.
- [client/src/components/TopBar.tsx](../../../client/src/components/TopBar.tsx): add three buttons that call back into App's view setter.

**No backend change.**

**Why grouped:** tabs without buttons (or vice versa) is unusable; splitting the PR creates a half-broken intermediate state.

## Step 3 — Alerts fetch + severe alert banner (idea #3, plus data plumbing for idea #2)

**Goal:** Fetch active NWS alerts for the configured point, expose them in the API response, and render a banner when active alerts exist.

**Server changes:**
- Add NWS `/alerts/active?point={lat},{lon}` to the normalizer's parallel fetch group.
- New cache TTL for alerts (~5 min — alerts change faster than forecasts and we want timely updates).
- Add `Alert` type to [shared/types.ts](../../../shared/types.ts) with at minimum: `id`, `event` (e.g., "Tornado Warning"), `severity` (Extreme/Severe/Moderate/Minor/Unknown), `headline`, `description`, `effective`, `expires`.
- Add `alerts: Alert[]` to `WeatherResponse`.

**Client changes:**
- New `<AlertBanner />` component, rendered conditionally above the TopBar when `alerts.length > 0`.
- Banner shows the highest-severity alert's headline. If multiple alerts active, show count (e.g., "3 active alerts — Tornado Warning until 9:45 PM").
- Banner is dismissible per-alert-id, persisted in `localStorage` so the same alert doesn't re-prompt every refresh. If a *new* alert appears, banner re-shows.

**Out of scope:** alert-driven color changes (that's Step 4). Banner only here.

**Why this is the most invasive step:** touches server (new fetch + cache), shared types (new shape), and client (new component, dismissal state). Worth landing alone before adding visual side-effects on top.

## Step 4 — Alert-driven UI color override (idea #2)

**Goal:** When an active alert exists, override the UI's base color scheme with a severity-mapped color.

**Color mapping (severity → color):**

| Alert event | Color |
|---|---|
| Tornado Emergency | Purple |
| Tornado Warning | Red |
| Severe Thunderstorm Warning | Orange |
| Tornado Watch / Severe Thunderstorm Watch | Yellow |
| Flood Warning / Flood Watch / Flash Flood | Green |
| (any other alert) | No color override; banner only |

When multiple alerts are active, the highest-severity color wins (Purple > Red > Orange > Yellow > Green).

**Implementation surface:**
- **Scoped CSS-var refactor:** convert *only* the colors that need to be theme-able (background tint, primary accent, banner border) into CSS variables. Everything else stays hardcoded — this is not a sweeping refactor.
- Compute highest-severity alert client-side from `alerts[]`.
- Apply a `data-alert-tier="purple|red|orange|yellow|green"` attribute on the root element, with CSS rules that swap the variables when the attribute is set.
- When no alert is active, no attribute is set; CSS vars hold their default values.

**Why bundle the CSS-var refactor here:** doing it as a separate "no-op refactor PR" feels like make-work; doing it here is when we first need it. Step 5's color picker will reuse the same vars without changes.

**Safety constraint (carried into Step 5):** Step 5's color picker palette MUST NOT include any color that visually resembles an alert tier (purple/red/orange/yellow/green). If a user's chosen base color matches an alert color, the alert visual signal is lost. See Step 5 for the curated palette approach.

## Step 5 — Settings gear (idea #6, both 6a and 6b)

**Goal:** Add a settings UI for unit and color preferences, persisted to `localStorage`.

**UI surface:**
- Gear icon in the TopBar (right side).
- Click opens a modal or drawer (final choice during implementation; modal is simpler).
- Two settings, two sections:
  - **Units:** °F (default) / °C toggle.
  - **Theme color:** four to six pre-coded color swatches.

**5a — °F/°C toggle:**
- Persisted to `localStorage` as `skyframe.units` (`"imperial" | "metric"`).
- Threaded through Current/Hourly/Outlook panels via React context to avoid prop-drilling.
- Conversion happens client-side at render time. The server response stays in imperial (no server changes).
- All temperature displays update: hero temp, "feels like", high/lows, hourly chart axis labels, dewpoint metric.
- Wind speed unit follows the same toggle (mph ↔ km/h). Pressure stays in inHg for both since metric pressure (hPa) is unusual for US users; revisit if user requests.

**5b — Curated color picker (palette, not free-form):**
- Pre-coded swatches set the *default* CSS vars introduced in Step 4. Step 4's alert override temporarily replaces them when an alert is active, then reverts when alerts clear.
- Persisted to `localStorage` as `skyframe.theme` (e.g., `"cyan" | "amber" | ...`).
- **Safety palette:** must avoid colors that resemble alert tiers (purple, red, orange, yellow, green). Proposed starter palette:
  - **Cyan** (current HUD default — closest to existing look)
  - **Blue** (deeper, less saturated)
  - **Teal** (cyan-green, distinct from flood-green)
  - **Magenta** (pink-leaning, distinct from tornado-purple)
  - **White / monochrome** (no color tint, neutral readout look)
- Final palette to be decided during implementation; the constraint (no alert-color overlap) is locked.

**Splittable:** if the PR feels too large, split into 5a (°F/°C) first and 5b (color picker) second. 5a is mechanically simpler; 5b reuses Step 4's CSS-var infrastructure with no new abstractions.

## Step 6 — Location change (idea #7) — deferred to its own brainstorm

**Status:** In v1.1 scope, but design happens in a separate brainstorming session before implementation.

**Open questions for that session:**
- True multi-location (save several locations, switch between them) vs. single configurable location?
- ZIP→lat/lon resolution: embed a US ZIP database (large), call NWS `/points/{zip}` (doesn't exist), use a third-party geocoder (violates "no third-party services" hard rule), or have the user enter lat/lon directly?
- What invalidates: NWS point metadata cache (currently 24h), station selection, timezone, alert point parameter — all change per location.
- UX: settings field, separate location switcher, autocomplete?

**CLAUDE.md update:** when Step 6 ships, the "Locked technical decisions" section moves from "hardcoded" to "user-configurable, default = {CITY} {ZIP}".

## Cross-cutting notes

**Testing:** existing Vitest setup covers normalizer, cache, and icon mapping. Add tests for:
- Alert fetch + cache (Step 3)
- Alert severity ranking (Step 4)
- Unit conversion functions (Step 5a)
- localStorage read/write helpers (Step 5)

**Dependencies:** prefer to add nothing. The settings modal can be a plain React component; no UI library needed. localStorage is built-in. No new server packages required.

**Out of scope for entire v1.1:**
- Mobile-specific layout changes
- Multiple monitors / multi-window support
- Push notifications
- Historical weather lookup
- Anything requiring a third-party service or API key
