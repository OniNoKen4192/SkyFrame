## 1. °F/°C Toggle

**What:** Click the hero temperature to toggle temperature units globally.

**Behavior:**
- Hero temperature in CurrentPanel becomes clickable
- Cursor pointer + subtle hover glow as affordance
- Click toggles between °F (default) and °C
- All temperatures across all panels update immediately: hero, feels-like, hourly chart axis, outlook range bars and labels, dew point, trends
- Persisted to `localStorage` as `skyframe.units`
- Conversion happens client-side; server continues to serve °F

**Not included:** settings gear, theme color picker. If enough settings accumulate later, the gear comes back.

## 2. Icon Expansion

**What:** Expand the icon set to cover all 34 NWS weather states. See `docs/icon-gaps.md` for the gap list.

**Process:**
1. User produces new SVG symbols for the semantic gaps
2. Symbols added to `client/icons.svg`
3. New `IconCode` values added to `shared/types.ts` matching the symbols
4. `server/nws/icon-mapping.ts` updated to map all 34 NWS slugs to the appropriate icon (new icons where available, fallback to existing icons otherwise)

**Daily icon override (Cedar Rapids bug):**

The current bug: a day with 90% precip probability shows a sun icon if NWS chose "partly sunny" as the period's dominant condition. Fix: override the NWS period icon when precip probability is high.

- Add a `HIGH_PRECIP_THRESHOLD = 50` constant in `icon-mapping.ts`
- For daily periods only: if `precipProb >= HIGH_PRECIP_THRESHOLD` and the NWS-mapped icon is not already a precip icon, upgrade it based on context (rain/thunder/snow as appropriate)
- Mirror of the existing hourly `PRECIP_PROB_THRESHOLD = 30` downgrade logic, inverted
- Hourly behavior unchanged

**Centered sun/moon:**

When the current conditions icon is clear sky (sun or moon, no clouds), center the icon in the CurrentPanel icon area instead of the current top-right positioning. CSS-only change driven by a data attribute on the icon container.

## 3. Advisory Tiers

**What:** Expand the alert system from 11 tiers to 13. Nothing from NWS gets silently dropped anymore.

> **Tier-count note (2026-04-18):** PR #5 added `tornado-pds` (rank 2) and `tstorm-destructive` (rank 4), so the existing tier count is now 11 (was 9 when this spec was first drafted). The two new advisory tiers below append at ranks 12 and 13.

**New tiers:**

| Tier | Rank | Color | Banner style | Theme override? | Dismissible? |
|---|---|---|---|---|---|
| `advisory-high` | 12 | Honey-orange `#ffaa22` | Hazard stripes (same pattern as severe tiers) | No | Yes |
| `advisory` (catch-all) | 13 | Base cyan (`--accent`) | Hazard stripes (same pattern, just in cyan) | No | Yes |

> **Color + stripes finalized 2026-04-18 via [docs/mockups/alert-tier-colors.html](../../mockups/alert-tier-colors.html):** Honey-orange `#ffaa22` chosen for `advisory-high` after visual comparison against five candidates. Original spec called for `advisory` to render as a "standard banner (no stripes)"; visual mockup showed the no-stripes version reads as broken layout (empty colored bands without animation), so `advisory` now uses stripes too. Visual hierarchy is signaled by color alone, not stripe presence.

**Known events mapped to `advisory-high`:**
- Wind Advisory
- Winter Weather Advisory
- Dense Fog Advisory
- Wind Chill Advisory
- Freeze Warning
- Freeze Watch
- Frost Advisory
- (extensible — add Map entries as needed)

**Catch-all behavior:**
- Any NWS event name not matched in the `EVENT_TO_TIER` map falls to `advisory`
- Classification logic returns `advisory` for unknowns instead of dropping them
- Update notifications (Section 8) also surface through this tier

**Implementation notes:**
- Two new entries in `TIER_RANK`
- New known advisory events in `EVENT_TO_TIER`
- Modify alert classification to return `advisory` instead of null/drop for unmatched events
- CSS: `[data-alert-tier="advisory-high"]` reuses the existing hazard stripe keyframe animation, just with yellow-orange colors
- CSS: `[data-alert-tier="advisory"]` uses the standard banner markup in base cyan
- Neither tier modifies `--accent` / `--accent-rgb` on the root — UI theme stays whatever it was (cyan default, or a higher-severity alert's color if one is active)

## 4. Alert Detail Terminal Modal

**What:** Click an alert event name in the expanded alert list to open a terminal-style modal with the full NWS alert text.

**Trigger:**
- In the expanded alert list (existing `AlertBanner` expanded state), each alert's event name becomes a clickable text element: underline + cursor pointer + hover state
- Click opens the terminal modal for that specific alert

**Modal visual design:**
- Fixed size: ~70-80% viewport width, centered vertically and horizontally
- Dark overlay (`rgba(0,0,0,0.5)`) dims the dashboard behind the modal
- Modal background: slightly lighter than the dashboard to separate visually
- Border in the alert's tier color with a subtle outer glow
- Monospace font throughout — HUD-consistent terminal aesthetic

**Title bar:**
- Left: `▲ {EVENT NAME}` (uppercase, tier color, bold)
- Right: issued timestamp (e.g., `12:34 PM CDT`) + close × button
- Bottom border in tier color at 30% opacity

**Body:**
- Meta line: `ISSUED {time} · EXPIRES {time} · {AREA}` — bright enough to read clearly (tier color at full opacity, not dimmed)
- Full NWS `description` text, rendered as paragraphs with preserved line breaks
- `HAZARD...`, `SOURCE...`, `IMPACT...` lines highlighted in the tier color (regex match on line prefix)
- Raw NWS text — no reformatting, no markdown processing
- Scrollable if content exceeds modal height

**Close:** × button in title bar, or click outside the modal

**Data flow:**
- NWS `/alerts/active` endpoint already returns a `description` field per alert
- Current normalizer discards it; needs to be retained
- Add `description: string` and `issuedAt: string` fields to the `Alert` type in `shared/types.ts`
- Normalizer passes these through to the client

**Reusable component:** The modal is the same primitive used for the NWS narrative forecast (Section 5). Build once, configure via props (title text, body text, tier color for border/accents).

## 5. NWS Narrative Forecast Popup

**What:** Reuse the terminal modal to display NWS's human-written daily forecast narrative.

**Triggers:**

**A) TopBar button** — a terminal icon (`▸` or similar) in the TopBar, right side. Click opens today's narrative.

**B) Clickable day in 7-day outlook** — day labels (e.g., "FRI · APR 17") become clickable text elements. Click opens that specific day's narrative.

**Data source:**
- `/gridpoints/{office}/{gridX},{gridY}/forecast` already returns a `detailedForecast` field per period (both day and night periods)
- Currently dropped by the normalizer; must be retained
- Add `detailedForecast: string` field to `DailyPeriod` in `shared/types.ts`
- For day/night pairs, the API returns them as separate consecutive periods

**Modal content:**
- Title bar: `▸ FORECAST · {DAY NAME} {DATE}` (e.g., `▸ FORECAST · FRI APR 17`) or `▸ FORECAST · TODAY` from the TopBar button
- Right side of title bar: NWS forecast issuance timestamp + close ×
- Body: shows both day and night narratives for the selected date, labeled and stacked:
  - `{DAY NAME}` section header
  - Narrative paragraph
  - `{DAY NAME} NIGHT` section header
  - Narrative paragraph
- Monospace text, no reformatting

**Visual distinction from alert modal:**
- Accent/border color stays base cyan (this isn't an alert)
- No tier-specific color overrides
- Same layout, same close behavior

## 6. Alert Sounds

**What:** Synthesized audio for top-severity alerts when a new alert appears.

**Behavior:**

| Tier | Sound |
|---|---|
| `tornado-emergency` | Repeating pulse until acknowledged |
| `tornado-pds` | Repeating pulse until acknowledged |
| `tornado-warning` | Repeating pulse until acknowledged |
| `tstorm-destructive` | Repeating pulse until acknowledged |
| `severe-warning` | Single play |

**Why `tstorm-destructive` is in the repeating group:** NWS standing guidance is to treat destructive thunderstorm warnings (`thunderstormDamageThreat=DESTRUCTIVE`) with tornado-level urgency — same shelter-in-place response. The audio behavior mirrors that operational guidance.

**Acknowledgment (global, not per-alert):**

Any of the following actions silences ALL currently-repeating sounds, regardless of which alert triggered them:

- Clicking any alert banner
- Opening any alert's detail modal
- Dismissing any alert

Rationale: an active tornado-class alert is a high-stress moment. Forcing the user to silence each alert individually adds friction at exactly the wrong time. One interaction = quiet.

**Acknowledgment persistence:**

- Acknowledged alert IDs are stored in `localStorage` (similar pattern to dismissal tracking)
- If the app reloads while an acknowledged alert is still active, the sound does **not** replay
- Acknowledgment state is cleared for an alert ID when that alert is no longer present in the NWS feed (so a re-issued alert with a new ID would re-trigger normally)

**Audio implementation:**
- Web Audio API synthesized tones — no audio files, no licenses, no external dependencies
- Sharp, attention-getting tone; exact sound to be tuned during implementation
- Not a continuous siren — a pulsing beep pattern for repeating alerts

**Trigger logic:**
- Fires only on *new* alerts (not seen in prior polls)
- Track played alerts by ID in a Set (similar pattern to dismissal tracking)
- If app starts up with an active top-tier alert already present and not previously acknowledged, it plays once
- Persistent identical alerts across poll cycles do not re-trigger

**Mute:** No mute toggle in v1.2. Can be added later if sounds become disruptive (would live in the future settings panel).

**Excluded tiers:** All other tiers (`blizzard`, `winter-storm`, `flood`, `heat`, `special-weather-statement`, `watch`, `advisory-high`, `advisory`) are silent.

## 7. GPS Autodetect

**What:** "Use my location" button in the existing location setup modal.

**Behavior:**
- Button added to `LocationSetup` modal alongside the existing ZIP and lat/lon inputs
- Click → browser Geolocation API prompts the user for permission
- On success: receives lat/lon, feeds into the same `/api/setup` flow the manual lat/lon path uses
- On permission denied or other failure: show a small inline error, user can still use ZIP or manual entry

**Localhost gating:**
- On component mount, read `window.location.hostname`
- Enabled: `localhost`, `127.0.0.1`
- Disabled: any other hostname — button greyed, tooltip: "GPS requires localhost"
- Rationale: browsers block Geolocation over non-HTTPS/non-localhost origins; this prevents an apparently-functional button that silently fails

**Server changes:** none. Location data flows through the existing `/api/setup` handler.

## 8. Update Notifications

**What:** Surface new GitHub releases as a catch-all advisory.

**Check schedule:**
- At server startup
- At local midnight daily
- No per-request or per-poll checks

**Implementation:**
- Server-side (not client) — keeps all outbound requests funneled through the server
- Unauthenticated GET to `https://api.github.com/repos/{owner}/{repo}/releases/latest`
- Compare response `tag_name` against the app version read from `package.json`
- If the release is newer: create a synthetic alert and feed it through the existing alert pipeline

**Synthetic alert shape:**
- Event: `"Update Available"`
- Description: short message with the new version number (e.g., "SkyFrame v1.3 is available") and a pointer to the release URL
- Tier: `advisory` (catch-all, base cyan banner)
- Issued timestamp: now
- Expires: far-future (never auto-expires until dismissed or version matched)
- Dismissible like any other alert

**Behavior:**
- Appears in the alert banner alongside any real weather alerts, sorted by rank (so it never preempts a real weather alert)
- Clickable (like other alerts) — opens the terminal detail modal showing the GitHub release notes body
- Dismissible; dismissal persists in localStorage like other alerts

**Failure handling:**
- GitHub unreachable, rate-limited, or API response malformed → silent skip, no error surfaced to UI
- Next check proceeds on schedule
- Rate limit for unauthenticated requests is 60/hour; 2 checks/day is nowhere near

**Privacy:**
- GitHub sees the server's IP address and User-Agent string (same exposure surface as NWS already sees for forecast requests)
- No app-specific data, no tracking identifiers, no usage metrics
- Pull-based check, not a push report — consistent with the "no telemetry" hard rule
