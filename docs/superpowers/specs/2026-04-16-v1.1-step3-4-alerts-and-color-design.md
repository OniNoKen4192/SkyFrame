**Date:** 2026-04-16
**Status:** Draft — pending user review
**Roadmap context:** [v1.1 roadmap](2026-04-15-v1.1-roadmap-design.md), Steps 3 + 4 (merged)

## Goal

Fetch active NWS alerts for the configured point, render an alert banner above the TopBar with hazard-stripe styling, and override the dashboard's accent color based on the highest-severity active alert. Steps 3 and 4 from the roadmap are merged because they share infrastructure (the same alert-tier mapping drives both the banner color and the UI color override) and the user signed off on shipping them as one PR.

## Color palette and severity ranking

Higher rank wins when multiple alerts are active. Banner shows the highest-ranked alert as the primary headline; UI accent color comes from the same tier.

| # | Tier (event names matched) | Color | Hex (starting value) |
|---|---|---|---|
| 1 | Tornado Emergency | Deep purple | `#9400D3` |
| 2 | Tornado Warning | Red | `#ff4444` |
| 3 | Severe Thunderstorm Warning (incl. Destructive Severe variant) | Orange | `#ff8800` |
| 4 | Blizzard Warning | Stark white | `#ffffff` |
| 5 | Winter Storm Warning | Blue | `#4488ff` |
| 6 | Flood Warning, Flash Flood Warning | Green | `#22cc66` |
| 7 | Heat Advisory, Excessive Heat Warning, Excessive Heat Watch | Red-orange | `#ff5533` |
| 8 | Special Weather Statement | Violet | `#ee82ee` |
| 9 | Tornado Watch, Severe Thunderstorm Watch | Yellow | `#ffdd33` |

**Ranking rationale:** all "imminent severe weather" warnings (1-6) rank above heat-related and informational alerts (7-8); watches sit lowest because NWS convention treats "Watch" as "possible" and "Warning" as "imminent." Hex values are starting points and trivially tuneable per-tier via CSS variables.

Each tier also has a darker stripe variant (used in the hazard-stripe pattern of the banner) — derived in CSS by applying `filter: brightness(0.6)` to the base color, or hand-tuned in the per-tier CSS rules. Implementation will pick whichever looks better on first render.

**Tier mapping is exact-match on the NWS `event` field.** Any NWS alert whose `event` value is not in the table above is **dropped silently** (no banner, no UI change). This intentionally excludes Wind Advisory, Air Quality Alert, Frost Advisory, Dense Fog Advisory, and similar non-immediate-threat alerts. They're noted as v1.2+ candidates in [docs/userInput/v1.2 ideas.txt](../../userInput/v1.2%20ideas.txt).

## Server changes

### NWS alerts fetch
- Add `https://api.weather.gov/alerts/active?point=42.8939,-87.9261` to the normalizer's parallel fetch group.
- Cache TTL: **5 minutes** (alerts change faster than forecasts; 5min is a reasonable balance between freshness and not hammering NWS).
- Cache TTL constant lives in `server/config.ts` alongside the existing observation/forecast TTLs.
- Failure mode: if the alerts fetch fails (network, 5xx, malformed JSON), log it and return an empty `alerts: []` array. The rest of the response still works. A failed alerts fetch must NOT cause the entire `/api/weather` endpoint to fail.

### Shared types

Add to [shared/types.ts](../../../shared/types.ts):

```typescript
export type AlertTier =
  | 'tornado-emergency'
  | 'tornado-warning'
  | 'severe-warning'
  | 'blizzard'
  | 'winter-storm'
  | 'flood'
  | 'heat'
  | 'special-weather-statement'
  | 'watch';

export interface Alert {
  id: string;            // NWS alert id (used as dismissal key)
  event: string;         // raw NWS event name e.g. "Tornado Warning"
  tier: AlertTier;       // mapped severity tier
  severity: 'Extreme' | 'Severe' | 'Moderate' | 'Minor' | 'Unknown';
  headline: string;      // NWS-supplied human-readable headline
  description: string;   // longer text (used in expanded banner if user clicks the alert? defer)
  effective: string;     // ISO timestamp
  expires: string;       // ISO timestamp (used for "UNTIL HH:MM PM CDT" display)
  areaDesc: string;      // e.g. "Waukesha + Milwaukee" (NWS-supplied)
}

// Add to WeatherResponse:
//   alerts: Alert[];   // empty array if no active alerts or fetch failed
```

### Tier mapping logic

New module: [server/nws/alert-tiers.ts](../../../server/nws/alert-tiers.ts).

Exports `mapEventToTier(event: string): AlertTier | null` — returns the tier for a known NWS event name, or `null` if unmapped (caller drops the alert).

Exports `tierRank(tier: AlertTier): number` — returns a numeric rank (1 = highest) for severity comparisons. Lower number = higher severity. Used to pick the primary alert when multiple are active.

Exports a frozen `TIER_COLORS` map for use in CSS-variable injection (or, more practically, the colors live in CSS rules keyed by `[data-alert-tier="..."]` and this map is just for documentation/test purposes).

Tests live next to the module at `shared/alert-tiers.test.ts` (co-located, since the module is shared between server and client). Cover:
- Each NWS event name in the palette table maps to the expected tier
- Unknown event names return `null`
- `tierRank` orders tiers correctly (test the full ordering)

### Normalizer integration

In `normalizeWeather()`:
- Add the alerts fetch to the existing parallel `Promise.all`
- For each alert in NWS response, run through `mapEventToTier`. Drop nulls. Build `Alert[]` from the survivors.
- Sort `alerts[]` by `tierRank` ascending (highest severity first) so `alerts[0]` is the primary.
- Include in `WeatherResponse.alerts`.

If the alerts fetch rejects, catch and substitute `[]`. Set `meta.error = 'partial'` if it's the only thing that failed (using the existing partial-failure flag).

## Client changes

### Alert tier mapping module

Lives at **[shared/alert-tiers.ts](../../../shared/alert-tiers.ts)** so both server and client import the same module. The existing `shared/types.ts` is already imported by both sides via relative paths (`'../shared/types'` from server, `'../../shared/types'` from client) — no tsconfig path changes required.

### `AlertBanner` component

New component: [client/components/AlertBanner.tsx](../../../client/components/AlertBanner.tsx).

Props: `alerts: Alert[]`. Returns `null` if `alerts.length === 0`.

**Layout (collapsed, single alert):**
```
[/// stripes ///] ▲ TIER · HEADLINE · UNTIL HH:MM PM CDT [\\\ stripes \\\] [×]
```

**Layout (collapsed, multiple alerts):**
```
[/// stripes ///] ▲ N ACTIVE ALERTS · TIER · HEADLINE UNTIL HH:MM [\\\ stripes \\\] [▾] [×]
```
The `[▾]` toggles expansion. The `[×]` dismisses just the primary alert.

**Layout (expanded):**
```
[/// stripes ///] ▲ N ACTIVE ALERTS · TIER · HEADLINE UNTIL HH:MM [\\\ stripes \\\] [▴] [×]
       • Tornado Warning · until 9:45 PM CDT  (Waukesha + Milwaukee)
       • Severe T-Storm Warning · until 10:30 PM CDT  (Racine)
       • Flood Watch · until 6:00 AM CDT  (SE Wisconsin)
```
Bullet rows use a clean dark background (no stripes), tier-color glyph (`•`) per row, monospace time formatting.

### Banner styling

- Stripes: `repeating-linear-gradient` at 45° on the left, -45° on the right (mirrored, converging toward center). Stripe width ~10px. Two shades of the tier color: bright base + darker variant.
- Stripe animation: `background-position` keyframes, ~5s per stripe-width cycle, flowing inward (left stripes drift right; right stripes drift left). Disabled by `@media (prefers-reduced-motion: reduce)`.
- Stripes pause when expanded (animation stays present but paused via `animation-play-state`).
- Banner appears/disappears via snap (no slide-in / fade). Consistent with Step 2 tab switching.
- Banner sits above the TopBar's existing dashed bottom border. Banner has its own dashed bottom border below the stripe row to separate it from the TopBar.

### Dismissal behavior

- `[×]` dismisses the **currently primary** alert by NWS `id`.
- Dismissed ids stored in `localStorage` under key `skyframe.alerts.dismissed` (JSON array of strings).
- On render, banner filters out any alert whose id is in the dismissed set.
- If a NEW alert appears (id not in dismissed set), banner re-shows even if other alerts are dismissed.
- Dismissed list is bounded — only retain ids for alerts that are still active in the latest response. Stale ids get pruned on each successful fetch (so the list doesn't grow unbounded over weeks).

### UI color override

A scoped CSS-variable refactor — only the colors that need to be theme-able become variables. The rest stays untouched.

**Variables to introduce in `:root` of [client/styles/hud.css](../../../client/styles/hud.css):**

```css
:root {
  --accent: #00e5d1;            /* base cyan, used everywhere */
  --accent-glow-strong: rgba(0, 229, 209, 0.9);
  --accent-glow-soft:   rgba(0, 229, 209, 0.5);
}
```

Replace the hardcoded `#00e5d1` and `rgba(0,229,209,...)` instances throughout `hud.css` with `var(--accent)` and the glow variables. ESTIMATED ~25-30 references. Done as a careful global find-and-replace, verified by visual inspection (no behavior change at this stage).

**Tier-driven override:** `App.tsx` sets a `data-alert-tier="..."` attribute on the root `.hud-showcase` element when an alert is active.

```css
.hud-showcase[data-alert-tier="tornado-warning"] {
  --accent: #ff4444;
  --accent-glow-strong: rgba(255, 68, 68, 0.9);
  --accent-glow-soft:   rgba(255, 68, 68, 0.5);
}
.hud-showcase[data-alert-tier="severe-warning"] {
  --accent: #ff8800;
  /* etc. */
}
/* ... one rule per tier ... */
```

When no alert is active, no `data-alert-tier` attribute is set, and the variables hold their default cyan values. Switching tiers is instant (no transition).

**Out of scope for the override:**
- The dark background `#000d10` stays the same regardless of tier.
- The dim text opacity (used for non-accent labels) stays the same.
- The footer dot's `dot-error` class (red offline indicator) stays red regardless of alert tier — it's a connectivity status, orthogonal to alert color.
- The TopBar offline link (`LINK.OFFLINE`) stays its red glow regardless of alert tier — same reasoning.

## Risks / edge cases

- **NWS event name variance:** NWS occasionally adds new event types or tweaks naming. The exact-match approach drops anything new silently. If a new tier needs to be supported, it's a one-line add to `mapEventToTier`. The test suite catches the case where a previously-supported event name silently changes.
- **Multiple alerts of the same tier:** Sort by `tierRank` is stable; ties are broken by NWS-supplied order (which is usually most-recent-first). The primary banner shows whichever ends up first; `[×]` dismisses just that one and the next-in-list takes over.
- **Alert expires while displayed:** No special handling. The next normal poll (≤5min) will return a fresh `alerts[]` without the expired alert; banner updates on next render. Worst-case stale display: 5 minutes.
- **Dismissed-list growth:** Pruning on each successful fetch (only retain ids still present in the new `alerts[]`) prevents unbounded growth.
- **CSS-var refactor risk:** Replacing ~25-30 hardcoded color references is mechanical but error-prone. Approach: do it as its own commit BEFORE wiring any tier overrides, with no behavior change. Visual diff confirms no regression. Then add the per-tier rules in a follow-up commit.
- **Banner + offline state:** Both can be active simultaneously (alert active + connection lost). The TopBar's red `LINK.OFFLINE` styling is intentionally orthogonal to the alert color override so the user can still tell connectivity status apart from alert tier.
- **Tornado Emergency vs deep purple Special Weather Statement:** The deep purple (#9400D3) for TE and pale violet (#EE82EE) for SWS are visually different (deep saturated vs pastel), so they shouldn't be confused. Confirmed during palette discussion.
- **Heat Advisory vs Severe Thunderstorm Warning:** Red-orange (#ff5533) and orange (#ff8800) are similar. Distinguishability relies on the headline text in the banner. Acceptable since these alerts wouldn't typically co-occur (heat is summer-day weather, severe is convective storm).
